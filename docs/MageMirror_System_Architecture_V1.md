# 一、系统目标定义（System Goal）

项目名称：

MageMirror（魔镜 AI衣橱系统）

系统目标：

构建平台级 AI衣橱系统，支持大量用户，实现：

- AI衣物识别
- AI自动分类
- AI搭配推荐
- AI试穿能力（未来）
- AI消费入口（长期目标）

系统本质：

AI衣橱 ≠ 工具  
AI衣橱 = 数据入口系统 + AI推理系统 + 用户消费系统

# 三、系统入口层（Input Layer）

当前最关键入口：

# 手机入口（核心）

目标：

手机直接拍照 → 直接上传 → 自动入库

支持方式：

- 手机浏览器访问 Web 页面
- HTML5 Camera API
- 支持多图上传
- 支持批量提交

未来扩展入口：

```

手机 Web App
微信小程序
iOS App
Android App
商品链接导入

```

长期目标：

入口必须支持：

```

拍照录入
批量录入
商品链接录入

```

# 四、Web App 层（前端层）

主要职责：

提供用户操作入口。

核心功能：

```

拍照上传

# 五、API 网关层（Gateway Layer）

作用：

统一管理所有请求入口。

推荐组件：

```

Nginx

或

Traefik

```

主要职责：

```

HTTPS管理
请求转发
路径分发
负载均衡（未来）
访问控制（未来）

```

作用极其关键：

未来所有入口：

```

手机
网页
小程序
API调用

```

必须统一经过 Gateway。

---

# 六、FastAPI 主服务（核心服务层）

当前系统核心：

FastAPI Server

主要职责：

```

接收上传请求
管理用户数据
触发图片处理
调用AI识别
管理衣橱数据
返回结果

```

主要接口：

```

/upload-clothes
/get-wardrobe
/delete-clothes
/update-clothes
```

未来接口扩展：

```

/match-outfit
/recommend
/try-on
/import-product
```

---

# 七、上传服务（Upload Service）

职责：

处理所有图片上传逻辑。

处理流程：

```

接收图片
临时保存
触发压缩
触发处理

```

必须支持：

```

多图片上传
批量处理
统一命名
异常处理
```

建议文件命名方式：

```

UUID + 时间戳

```

例如：

```

cloth_20260326_uuid.jpg

```

---

# 八、图片处理服务（Image Processing）

当前核心组件：

rembg

主要职责：

```

背景去除
衣物抠图
图片裁剪
居中处理
尺寸标准化
格式统一

```

目标输出：

```

统一尺寸
统一比例
统一背景
统一居中
```

推荐标准：

```

尺寸：1024x1024
背景：透明
格式：PNG
```

未来扩展：

```

衣架自动识别
轮廓优化
衣物区域增强
```

---

# 十一、数据存储层（Storage Layer）

必须拆分两种存储：

# 1）结构数据（数据库）

推荐：

```

PostgreSQL

```

存储内容：

```

用户数据
衣物信息
分类数据
标签数据
记录数据
```

---

# 2）图片存储（对象存储）

推荐：

```

AWS S3

或

MinIO（本地）

```

存储内容：

```

原始图片
处理后图片
缩略图
标准图
```

---

# 十二、缓存层（Cache Layer）

未来必须存在。

推荐：

```

Redis

```

主要用途：

```

加速查询
缓存结果
减少数据库压力
管理会话
```

---

# 十三、Worker 任务层（非常关键）

这是未来系统稳定的关键。

不能缺失。

推荐技术：

```

Celery
或
RQ
或
Dramatiq
```

主要职责：

```

后台处理图片
执行AI识别
生成向量
批量任务处理
```

执行流程：

```

FastAPI 接收请求
→ 推送任务
→ Worker处理
→ 返回结果
```

这一步未来必须实现。

---

```

# 十八、当前阶段总结（关键）

当前系统阶段属于：

```

真实数据验证阶段

```

不是：

```

算法阶段

```

不是：

```

AI模型阶段

```

而是：

```

入口体验阶段

```

```

不是优化AI
而是打通手机入口

```
